# Assembly

🧩 Декларативный движок многоуровневой инициализации для построения иерархических систем из движков и логических единиц.

> Позволяет описывать сложную инициализацию системы **по уровням**, используя правила сборки, пути и контексты.  
> Подходит для построения протоколов, движков, фреймворков и DSL, где важен порядок, вложенность и контекст инициализации.

---

## ✨ Идея

`Assembly` — это **инициализационный конвейер**, который:
- Собирает систему **по уровням**
- Передаёт контекст от родителя к потомку
- Использует правила инициализации (`rules`)
- Автоматически разрешает пути модулей
- Поддерживает вложенные области и логические единицы

Проще говоря:

**Вы описываете структуру — Assembly собирает её.**

## 🧠 Основные понятия

| Термин                         | Описание                                                 |
| ------------------------------ | -------------------------------------------------------- |
| **Уровень инициализации**      | Шаг сборки системы (движки → логика → компоненты и т.д.) |
| **Rule (правило)**             | Описывает, как инициализировать и регистрировать уровень |
| **Engine (движок)**            | Основной элемент уровня                                  |
| **Logic (логическая единица)** | Вложенный элемент внутри движка                          |
| **Parent**                     | Родитель текущего уровня                                 |
| **Next**                       | Переход к следующему уровню инициализации                |

## 📥 Быстрый старт

### Инициализация Assembly

```js
const Assembly = require('assembly');

const Assembler = Assembly(
    require('./rules/engines'),    // Правило инициализации движков
    require('./rules/logic'),      // Правило инициализации логических единиц
    require('./rules/components'), // Правило инициализации компонентов
);
```

---

### Использование в классе

```js
class Protocol {
    constructor() {
        Assembler.engines(this, [
            'flow',
            'state',
            'crypto',
        ]);
    }
}

const protocol = new Protocol();
```

## 🧩 Как это работает

Assembly строит систему **сверху вниз**:

1. Выбирается область инициализации (`scope`)
2. Загружаются движки
3. Для каждого движка:
    - применяется правило текущего уровня
    - формируется контекст
    - выполняется переход на следующий уровень
4. Загружаются логические единицы
5. Процесс повторяется рекурсивно

---

## 📦 Использование Assembly в реальном движке

Assembly используется **внутри конструктора движка** как функция перехода к следующему уровню инициализации.

Движок **не знает о файловой структуре**,  
он лишь **декларирует свою внутреннюю композицию**.

---

### Пример движка

```js
module.exports = class FlowEngine {
    constructor(engine) {
    // Инициализируем движок
        engine(this);
        
    // Инициализируем список логических единиц
        engine.logic(
            'incoming', // Логика обработки входящих пакетов
            'outgoing', // Логика обработки исходящих пакетов
        );
    }
    
// Методы движка ...
    method1 = () => {}
    method2 = () => {}
    method3 = () => {}
};
```

---

### Что здесь происходит

- `engine` — это **функция перехода Assembly** к следующему уровню
- Вызов `engine(this)`:
    - Применяет `Rule.init`
    - Регистрирует движок через `Rule.register`
    - Подготавливает контекст для следующего уровня
- Вызов `engine.logic(...)`:
    - Активирует следующий уровень инициализации
    - Загружает логические единицы из соответствующей директории
    - Передаёт им новый контекст

---

### Ключевая идея

Движок **не управляет загрузкой компонентов напрямую и не импортирует свои логические единицы вручную**.

Вместо этого он:
- Описывает **структуру**
- Объявляет **уровни**
- Доверяет Assembly **сборку системы**

Фактически:

> **Движок — это декларация.**  
> **Assembly — это исполнитель.**

---

## 🧠 Почему это важно

Такой подход позволяет:
- Изолировать **логические единицы и компоненты** от структуры файлов
- Легко добавлять новые уровни инициализации
- Гарантировать порядок сборки
- Строить большие иерархические системы без связанного кода
- Использовать Assembly как DSL для архитектуры системы

## 🔧 API

`Assembly(...rules)`

Создаёт доменный инициализатор.

- `rules` — список правил инициализации по уровням

Возвращает Proxy-объект, через который происходит инициализация.

### Правило инициализации (Rule)

Каждое правило должно реализовывать следующий интерфейс:

```js
module.exports = {
    path(scopePath, unitPath) {},
    init(parent, current) {},
    register(parent, current, scopePath, unitPath) {},
};
```

### Методы:

- `path(scopePath, unitPath)` — Возвращает путь к логической единицы для загрузки
- `init(parent, current)` — Инициализирует текущую логическую единицу
- `register(parent, current, scopePath, unitPath)` — Регистрирует логическую единицу в родительском контексте

---

### 📐 Канонический шаблон Rule

В репозитории Assembly определён **официальный шаблон правила инициализации**,  
который отражает архитектуру и философию проекта:

📄 **[`/templates/rule.js`](./templates/rule.js)**

Этот шаблон:

- Фиксирует порядок `path → init → register`
- Документирует Rule как **алгоритм сборки**
- Служит единым стандартом для всех уровней инициализации
- Рекомендуется как основа для `engines`, `logic`, `components` и других уровней

Использование шаблона позволяет сохранять:
- Единый стиль кода
- Читаемость Rule-файлов
- Предсказуемость поведения Assembly

---

## 🧠 Философия Rule: порядок имеет значение

В `Assembly` правило инициализации (**Rule**) рассматривается не как набор методов,  
а как **алгоритм загрузки и сборки системы**.

Rule отвечает на вопросы **строго в следующем порядке**:

1. **Где лежит модуль?** → `path`
2. **Что с ним сделать при создании?** → `init`
3. **Как встроить его в систему?** → `register`

Это формирует базовый конвейер Assembly:

```
path → init → register
```

Философски это можно выразить так:

> **Сначала я знаю, что загрузить,  
> потом — как инициализировать,  
> и только потом — как подключить к системе.**

Именно поэтому в Rule порядок методов имеет значение и отражает реальный процесс
работы Assembly — от разрешения путей и загрузки модулей до их регистрации
в родительском контексте.
